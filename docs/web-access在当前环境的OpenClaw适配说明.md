## 三、它解决的核心问题

`web-access` 不是普通的“能搜网页” skill。

它真正补的是三类短板：

### 1. 联网策略层
在以下方式之间做场景选择：
- WebSearch
- WebFetch
- curl
- Jina
- CDP 浏览器

这点很重要，因为很多系统的问题不是“不能联网”，而是：
**不知道什么时候该用哪种联网方式。**

### 2. 动态网页 / 登录态层
它支持通过 CDP 连接用户浏览器，实现：
- 动态页面访问
- 页面内点击、滚动、提取
- 登录态站点访问
- 文件上传
- 截图
- 视频内容抽帧

### 3. 站点经验层
它支持按域名积累“站点经验”，也就是：
- 某站点什么方式最有效
- 什么 URL 结构靠谱
- 哪些坑不要踩

这一层对长期自动化很有价值，因为它天然适合积累“平台知识”。

## 五、在 OpenClaw 当前工作流里的适合用法

## 1. 适合优先使用的场景

### A. 网页调研与资料发现
适合：
- 搜某个主题最新进展
- 读取公开网页内容
- 对多个站点做对比调研
- 做热点信息发现

### B. 动态站点内容读取
适合：
- 静态抓取拿不到正文
- 页面需要滚动、点击、展开后才能看到内容
- 站点依赖前端渲染

### C. 登录态任务
适合：
- 访问用户已登录平台内容
- 创作者平台后台操作
- 需要浏览器上下文的页面流程

### D. 自媒体 MVP / 自动化体系中的潜在位置
对你当前在做的系统来说，它很适合未来放到：
- 热点发现
- 素材抓取
- 网页内容获取
- 站点访问 fallback 路由

特别是自媒体链路里，它会比“只会 requests + BeautifulSoup”更强。

## 六、当前不建议立刻重度依赖的地方

### 1. 不要一上来就把它设为所有联网任务的唯一入口
原因：
- 当前环境还没完成 CDP 验证
- 你现有体系里还有原生 `web_search` 能力
- 盲目切总入口，风险高

### 2. 不要默认把它绑定到高风险自动化操作
比如：
- 自动发平台内容
- 自动登录后执行敏感动作
- 依赖浏览器状态完成不可逆操作

原因：
- 还没做稳定性验证
- 浏览器自动化天然更脆弱

### 3. 不建议先做“全量接管”，而建议先做“分场景 fallback”
也就是说：
- 先把它当增强能力
- 不是先把它当唯一能力
