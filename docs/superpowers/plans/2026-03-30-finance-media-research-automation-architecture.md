# 财经自媒体与投研自动化系统总方案（v2）

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 设计一套同时覆盖“财经自媒体自动化/半自动化”和“金融投研自动化/半自动化”的统一工作流系统，兼顾成本、质量、可维护性与扩展性。

**Architecture:** 采用“Python 服务 + n8n 调度 + YMOS 投研闭环 + 云端 API 主脑 + 本地模型辅助”的混合架构。系统按数据采集层、规则处理层、AI 分析层、交付层、人机协同层、调度监控层分层；财经自媒体和投研共享基础设施，但分别保留独立流水线和质量闸门。

**Tech Stack:** Python、FastAPI、n8n、SQLite/PostgreSQL、ChromaDB、Ollama、Qwen2.5-7B、Qwen2.5-Coder、faster-whisper、YMOS、Gemini / GPT / DeepSeek API、图像生成 API、飞书/企业微信/Telegram 通知、Streamlit（MVP 审校页）、Prometheus/Grafana（正式环境可选）

## 0. 设计哲学：工程化思维 + 精准主义

本系统的设计遵循以下两条核心哲学，它们决定了所有架构选择、流程设计和实施优先级。

### 0.1 工程化思维：把内容和研究当作系统来构建

- **做财经内容**：不依赖灵感，而是建立固定模板、固定流程、固定选题库。像写代码一样批量产出高质量文案，用复用和参数化替代每次从零开始。
- **做交易与研究**：不凭感觉，只按规则执行。写清楚筛选条件、买入/卖出逻辑、止损止盈阈值。系统只做规则允许的事，不引入情绪波动。
- **高频、确定性、可模板化流程优先固化为代码**：凡是重复频率高、输入输出边界清晰、可验证可回归的环节，优先沉淀为规则、函数、模板、导出器和状态机，而不是每次都交给 AI 临时推理。
- **核心竞争力**：别人靠灵感，你靠系统。可复制、可回测、可优化。

### 0.2 精准主义：抠细节，求极致

- 数据更准：规则清洗 + 来源校验 + 人工抽查。
- 逻辑更硬：AI 辅助但由规则兜底，所有关键结论必须附带**依据、逻辑摘要、风险点与适用边界**。
- 观点更稳：高风险内容经人工审校 + 风险提示，不输出模糊或夸张表述。
- **对外标签**：理性、严谨、可落地、不忽悠。这在财经内容与投研领域极为稀缺。

### 0.3 PDCA 闭环（Plan-Do-Check-Act）

本系统所有任务流遵循 PDCA 循环：

- **Plan**：由 n8n 定时或事件触发任务计划，明确输入、输出、资源预算。
- **Do**：Python 服务执行规则 + AI 分析，记录所有中间状态。
- **Check**：任务状态机、死信队列、可观测性指标、人工审校结果、回流数据。
- **Act**：周度统计脚本分析反馈，调整 prompt、规则阈值、选题评分模型、筛选条件。

每一轮自动化都不是终点，而是下一次优化的起点。

## 1. 目标定义

本系统服务两个核心需求：

1. **财经自媒体自动化/半自动化**
   - 自动追踪热点并生成选题建议
   - 抓取爆款图文/视频并做转写、拆解
   - 基于选题自动生成高质量文案并配图
   - 可选导出到公众号、知乎等平台

2. **金融投研自动化/半自动化**
   - 自动采集公开财经网站数据：行情、财务指标、公告、新闻
   - 根据用户自定义规则筛选股票/标的
   - 利用 AI 做基本面分析、情绪解读和风险提示
   - 定时执行并推送日报/盘后报告

**总原则：**
- 确定性任务优先使用代码与规则引擎
- AI 主要承担摘要、归纳、解释、生成与多文档综合推理
- 默认采取**半自动化**，尤其是外发内容与投资建议输出，保留人工审校阀门
- 优先做最小可用闭环，再做降本与自动化增强
- 所有自动化都必须纳入预算、可观测性、熔断与人工接管框架

## 2. 总体架构结论

推荐采用：

### **混合架构：API 为主，本地模型为辅**

不是全 API，也不是本地模型主导。

#### 云端 API 负责：
- 长文档/多文档深度分析
- 重要投研结论与终稿报告生成
- 高质量中文写作终稿
- 复杂多模态理解
- 配图生成

#### 本地模型负责：
- 高频低价值批处理
- 热点初筛/聚类/标签归类
- 私有语料 RAG 召回后的初稿改写
- 批量文案提纲生成
- 成本敏感的日常清洗与预处理

#### 代码与工作流系统负责：
- 爬虫/采集/解析
- 数据清洗、标准化、去重
- 指标筛选与规则判断
- 定时调度、错误处理、重试、通知推送
- 任务状态管理、预算控制、日志与可观测性

## 3. 架构硬约束（必须遵守）

1. **n8n 只做编排，不直接执行本地脚本。**
2. **所有核心能力必须通过 Python HTTP API 暴露。**
3. **MVP 只允许先闭环一条主线：要么先做自媒体选题列表，要么先做投研筛选。**
4. **所有异步重任务必须具备任务状态、自动重试、死信队列（DLQ）。**
5. **本地 GPU 任务必须受控并发，并按时段调度，避免显存溢出。**
6. **云端深度分析与本地批处理必须职责分层，不得混用。**
7. **所有云端计费调用必须具备预算、token 上限、重试熔断与人工解锁。**
8. **系统必须预留 metrics / logging / trace_id 能力。**
9. **MVP 必须提供极简审校界面，不允许人工直接翻数据库完成审校。**
10. **系统必须记录反馈回流字段，用于周度统计优化。**

## 4. 系统分层设计

### 4.1 数据采集层

负责从公开渠道抓取和整理原始信息，不依赖 AI。

#### 财经自媒体数据源
- 新浪财经热点/快讯
- 东方财富公开资讯页
- 同花顺公开页面
- 华尔街见闻、RSS 聚合源
- B 站 / YouTube / 抖音财经频道元数据
- 人工补充导入的公众号/专栏链接

#### 投研数据源
- Tushare
- AkShare
- 巨潮资讯公告
- 交易所公告
- 东方财富、新浪财经财务页
- Yahoo Finance / Finnhub / SEC EDGAR（可选扩展）
- 宏观与链上数据（可后续扩展）

#### 产出
- 标准化原始数据表
- 原文/正文/转写文本
- 元数据：来源、抓取时间、内容类型、标的映射、热度指标

#### 联网采集规范
- **复杂联网任务**（动态页面、登录态、反爬站点、真实浏览器交互）优先通过 OpenClaw 的 `web-access` 技能执行。
- **简单静态公开内容**（RSS、公开 HTML、稳定 API）允许直接使用 `httpx` / `feedparser` / 规则抓取，以降低复杂度与时延。
- `web-access` 提供统一的联网路由策略（静态抓取 → Jina → CDP）以及站点经验沉淀能力，应作为复杂网页采集的默认增强层。

---

### 4.2 规则处理层

负责确定性逻辑，避免把不该由 AI 处理的内容交给模型。

#### 职责
- 网页正文抽取
- 去重与相似度判重
- 热点聚类
- 股票代码/行业映射
- 财务指标筛选
- 公告分类
- 风险词过滤
- 任务路由
- 上下文裁剪与分块
- token 预算预估

#### 原则
- 能用规则完成的，不先用 AI
- 高频、确定性、可模板化流程必须优先下沉到规则处理层，不得默认上推到 AI 分析层
- AI 在规则之后，提高质量而不是替代基础流程

---

### 4.3 AI 分析层

分为两级：本地模型层、云端模型层。

#### 本地模型层
适合：
- 批量初筛
- 低成本摘要
- 选题标签生成
- 风格对齐初稿
- 批量热点压缩
- 自媒体素材结构拆解（第一轮）

#### 云端模型层
适合：
- 财报/公告/新闻/财务的综合研判
- 投资建议报告终稿
- 自媒体高质量终稿
- 多模态理解与配图提示词优化
- 高风险内容的最终审校增强

#### AI 层边界
AI 分析层不负责执行可由规则、模板、脚本稳定完成的标准化流程，例如字段提取、结构归一、固定格式导出、基础打分、模板装配、状态流转等。

---

### 4.4 交付层

#### 财经自媒体交付物
- 每日热点选题清单
- 爆款拆解卡片
- 文案草稿 / 终稿
- 封面图 / 配图
- 可导出的公众号 / 知乎 Markdown/HTML 稿件

#### 投研交付物
- 日报 / 盘后报告
- 筛股结果清单
- 单标的基础面摘要
- 投资雷达 / 策略分析 / 持仓收口
- Dashboard / HTML 报表 / Markdown 报告

---

### 4.5 人机协同层

#### 核心原则
- 所有高风险输出默认进入审校环节
- 人工审校必须基于页面化视图，而不是数据库或日志
- 人工反馈必须结构化写回系统，用于后续优化

#### MVP 阶段最低要求
- 提供极简审校页（Streamlit / 轻量 Web 页面）
- 支持查看待审任务列表
- 支持查看草稿/摘要/来源
- 支持操作：采纳、驳回、重生成、补充备注、导出

---

### 4.6 调度与监控层

#### 调度中心：n8n
职责：
- 定时触发
- 任务编排
- 调用 Python HTTP API
- 调用本地模型 API（Ollama）
- 调用云端模型 API
- 推送结果和报警

#### 监控与恢复
- 错误捕获
- 失败重试
- 超时控制
- 任务状态持久化
- 成功/失败通知
- 熔断状态管理

#### 原则
- n8n 负责“交通调度”，不承载复杂业务逻辑
- 复杂逻辑全部放在 Python 服务中

## 5. Python 服务接口化设计

### 5.1 设计原则
- Python 服务是业务能力承载层
- n8n 只能通过 HTTP 调用能力，不直接运行脚本
- 轻任务可同步返回，重任务必须异步化

### 5.2 推荐接口风格
#### 同步接口
- `POST /media/topics/generate`
- `POST /research/screen/preview`
- `POST /llm/local/summarize`

#### 异步接口
- `POST /tasks`
- `GET /tasks/{task_id}`
- `POST /tasks/{task_id}/retry`
- `POST /tasks/{task_id}/cancel`

### 5.3 推荐业务接口示例
- `POST /media/topics/generate`
- `POST /media/materials/analyze`
- `POST /media/drafts/generate`
- `POST /research/screen/run`
- `POST /research/report/generate`
- `POST /review/items/{id}/approve`
- `POST /review/items/{id}/reject`
- `GET /metrics`

### 5.4 OpenClaw 技能集成规范
- Python 服务调用 OpenClaw 技能应统一通过受控封装执行，不允许在业务代码中散落临时命令调用。
- 推荐两种集成方式：
  1. `subprocess.run([...])` 调用 `openclaw` 命令
  2. 通过 OpenClaw Gateway 的统一 HTTP 端点调用
- 所有技能调用必须设置超时（默认 30 秒）与失败处理策略（遵循第 11 章）。
- 对高频技能（如 `web-access`），应封装为 Python 函数或服务适配器，内部统一处理调用、解析、超时、重试与错误映射。
- MVP 阶段若只处理 RSS 与公开静态页面，可不强制接 skill；一旦进入复杂网页场景，应优先切换到 skill 适配器。

## 6. 两条业务流水线设计

## 6.1 财经自媒体流水线

### Step 1：热点采集
输入：新闻流、热搜榜、社交平台热点、指定账号更新
输出：原始热点池

### Step 2：热点清洗与聚类
动作：去重、分类、主题归并、热度评分
优先：规则 + 本地模型
说明：标题规范化、去重、相似聚类、来源统计、时间衰减等基础步骤优先由代码完成；AI 只补充命名、标签或高层概括。
输出：候选热点主题

### Step 3：选题评估
采用“双层结构”：基础分由规则引擎计算，AI 只对高分候选做角度生成与少量重排序。

评分维度：
- 热度
- 时效性
- 与账号定位相关性
- 重复度
- 潜在传播性
- 风险等级

输出：
- 今日快评题
- 深度分析题
- 情绪驱动题
- 高质量候选题写入固定选题库 `topic_library`，供后续复用、参数化微调与周度回流分析

### Step 4：素材抓取与拆解
- 网页正文抽取
- 视频下载/转写
- 提取 hook、结构、冲突点、CTA、情绪点

### Step 5：文案生成
拆成三阶段：
1. 角度与提纲
2. 初稿
3. 终稿润色

### Step 6：配图生成
- 根据选题与平台模板出封面图/配图
- 图像走云端 API

### Step 7：审校与导出
- 人工确认后导出
- 默认不直接自动发布

---

## 6.2 金融投研流水线

### Step 1：数据采集
- 行情
- 财务指标
- 公告
- 新闻
- 资金流 / 板块强弱（可选）

### Step 2：规则筛选
- 例如：低 PE、高增长、高 ROE、公告风险过滤、行业趋势条件
- 所有基础筛选条件（估值、增速、公告窗口、行业映射、资金流阈值、异常波动阈值）均优先由规则引擎完成
- 这一步只用代码与规则引擎

### Step 2.5：交易规则引擎（Phase 2 扩展）
对筛选出的标的，根据预设条件（如波动率、最大回撤、支撑阻力位）自动计算建议止损/止盈价位，并生成风控建议卡片。

约束：
- 不自动下单
- 只辅助人工决策
- 与基本面筛选结果分层保存
- 止损、止盈、仓位建议、波动率约束、最大回撤阈值等高频确定性计算优先固化为规则函数，不依赖 AI 临时推导

### Step 3：批量初评
对筛出的候选标的生成：
- 主营业务摘要
- 财务改善/恶化趋势
- 公告关键信号
- 近期情绪方向
- 基础风险提示

### Step 4：重点标的深挖
对 Top N 标的做：
- 多文档综合分析
- 基本面逻辑归纳
- 情绪与预期差判断
- 风险点拆解
- 应对预案

### Step 5：接入 YMOS 闭环
继续使用 YMOS 既有体系：
1. 市场洞察
2. 投资雷达
3. 策略分析
4. 持仓收口

### Step 6：定时推送
- 每日盘后
- 关键波动时手动触发
- 输出到 Markdown / HTML / 通知渠道

## 7. MVP 拆分策略

### 核心原则
**MVP 不并行做两条全功能主线。**
只允许先闭环一个最小业务链路。

### 路线 A：先做自媒体选题列表 MVP
最小闭环：
1. 抓热点
2. 聚类去重
3. 生成选题列表
4. 进入审校页
5. 导出 Markdown

### 路线 B：先做投研筛选 MVP
最小闭环：
1. 拉行情 / 财务 / 公告
2. 规则筛股
3. 输出候选标的
4. 生成简版分析
5. 输出日报

### 选择建议
- 若目标是尽快服务现有投研工作流：优先路线 B
- 若目标是最快做出轻量可视成果：优先路线 A

## 8. 模型与算力分工

## 8.1 是否需要本地模型

### 结论
**需要，但不应作为主脑。**

本地模型的价值在于：
- 降本
- 批处理
- 私有风格与私有知识库利用
- 高频低价值任务内循环

云端模型的价值在于：
- 长上下文能力
- 深度综合推理
- 终稿质量
- 多模态能力
- 更稳定的高质量结果

---

## 8.2 结合“至强 E5 + RTX 3060 12GB”的推荐

### 推荐本地模型
#### 1. Qwen2.5-7B-Instruct
适用：
- 热点分类
- 选题提纲
- 风格对齐初稿
- 简要摘要
- 批量标签任务

#### 2. Qwen2.5-Coder 7B
适用：
- 爬虫/ETL/自动化脚本开发与维护

#### 3. faster-whisper + large-v3
适用：
- 视频和音频转写

### 不建议作为主力生产模型的方向
- 14B 级本地模型承担主力投研终稿
- 本地模型承担超长财报综合分析
- 本地模型承担高风险对外发布内容的最终质量把关

原因：
- 3060 12GB 可以尝试更大模型，但长期稳定性、速度、批处理效率不理想
- 生产系统更看重稳定和吞吐，不是“能跑就行”

## 8.3 本地与云端调用决策表

### 使用本地模型的情况
- 批量 100~1000 条内容做初筛
- 简单摘要、标签、改写、提纲
- 历史文章风格 RAG 初稿
- 高频低成本处理
- 隐私敏感但推理强度不高的内容

### 使用云端 API 的情况
- 财报/公告/新闻综合推理
- 投资建议报告终稿
- 高质量自媒体外发终稿
- 多模态任务
- 复杂中文表达润色
- 高风险内容的最终审校

## 8.4 硬件调度策略（GPU / 显存保护）

### 设计目标
在 3060 12GB 环境下，避免本地模型、转写、Embedding 等任务并发抢占显存造成抖动、超时或崩溃。

### 任务队列分类
- **GPU 队列 A：本地 LLM 文本任务**
- **GPU 队列 B：Whisper 转写任务**
- **GPU 队列 C：Embedding / RAG 建库任务**
- **图像任务优先走云端 API**，默认不占用本地 GPU

### 运行约束
- 同一时刻只允许 1 个重 GPU 任务
- GPU worker 并发上限默认为 1
- CPU 任务可并发执行
- 白天优先轻量分析；夜间安排批量转写、建库与低优先级重任务

### 推荐分时策略
- **白天**：热点采集、规则筛选、轻量本地摘要
- **盘后**：投研筛选、候选分析、YMOS 闭环
- **夜间**：视频转写、RAG 重建、批量历史分析

## 11. 错误处理、重试与死信队列

### 11.1 任务状态机
每个任务至少具备以下状态：
- `pending`
- `running`
- `succeeded`
- `failed_retryable`
- `failed_terminal`
- `dead_lettered`

### 11.2 自动重试
对可重试错误使用指数退避：
- 第 1 次失败：30 秒后
- 第 2 次失败：2 分钟后
- 第 3 次失败：10 分钟后
- 超过阈值：进入 DLQ

### 11.3 可重试错误
- 网络超时
- API 429 / 限流
- 临时 5xx
- 本地模型短时无响应
- 源站短时不可用

### 11.4 不可重试错误
- 参数校验错误
- 认证失败
- 数据格式错误
- 缺失关键字段且无法修复
- 业务规则冲突

### 11.5 死信队列（DLQ）要求
DLQ 至少记录：
- 任务类型
- 输入参数
- 失败原因
- 重试次数
- 最后错误栈
- 创建时间
- 来源工作流

### 11.6 人工处理能力
DLQ 中的任务必须支持：
- 查看详情
- 一键重试
- 修改参数后重放
- 标记忽略

## 12. 成本控制与熔断机制

### 12.1 设计目标
防止因死循环、异常重试、超长上下文、恶意调用或配置错误导致云端 API 成本失控。

### 12.2 日预算控制
建议按模型或任务线设置单日预算，例如：
- `gpt-4o-mini`：$2 / day
- `gemini` 深度分析：$3 / day
- 图像 API：$1 / day
- 自媒体总预算：$3 / day
- 投研总预算：$5 / day

### 12.3 单任务上限
- 限制单任务最大 token 输入量
- 限制单任务最大 token 输出量
- 超限时执行：截断、分块或拒绝执行

### 12.4 重试熔断
- 同一任务因限流/超时失败超过 3 次，不再自动重试
- 直接进入 DLQ，并推送成本/稳定性告警

### 12.5 自动降级
当预算或模型稳定性触发阈值时：
- 非核心任务切换到便宜模型或本地模型
- 非必要任务直接暂停
- 核心任务等待人工确认

### 12.6 人工解锁
熔断触发后：
- 发送飞书 / 企业微信成本告警
- 标记系统状态为 `circuit_open`
- 需人工确认后恢复

### 12.7 AI 调用必要性治理
降本的第一优先级不是更便宜的模型，而是减少不必要的模型调用。所有可规则化、模板化、函数化的步骤，应优先消灭 AI 调用。

系统应持续统计各任务流中的 AI 调用次数、token 消耗、耗时与产出价值，用于识别“高消耗但低必要性”的 AI 环节。

每周 review 时，应输出“AI 调用重构优先级清单”，优先评估以下内容是否可下沉为规则、模板、脚本或状态机：
- 聚类与基础打分
- 固定格式导出
- 结构化字段提取
- 固定骨架报告生成
- 状态流转与模板装配

## 13. 可观测性设计（Metrics / Logging / Tracing）

### 13.1 MVP 阶段最低要求

- 所有任务和 API 调用带 `trace_id`

- `task_retry_total`（Counter）
- `api_call_duration_seconds`（Histogram）
- `api_call_total`（Counter，按模型名）
- `tokens_input_total` / `tokens_output_total`（Counter）
- `cost_estimate_usd_total`（Counter）
- `dead_letter_queue_size`（Gauge）
- `gpu_memory_used_bytes`（Gauge）
- `queue_length`（Gauge）
- `circuit_breaker_open_total`（Counter）
- `ai_call_count_by_flow`（Counter）
- `ai_tokens_by_flow`（Counter）
- `ai_cost_by_flow`（Counter）
- `ai_latency_by_flow`（Histogram）

### 13.3 日志要求
采用结构化日志，最少字段：
- `timestamp`
- `trace_id`
- `job_id`
- `task_type`
- `model_name`
- `status`
- `latency_ms`
- `retry_count`
- `estimated_cost`

### 13.4 AI 调用必要性指标

- 高价值 + 低成本
- 高价值 + 高成本

## 15. 反馈回流与持续优化机制

### 15.1 目标
让系统能基于实际采用结果和后验表现做简单统计优化，而不是长期停留在静态 prompt 和固定规则。

### 15.2 自媒体最小回流字段
- `selected`
- `published`
- `channel`
- `manual_score`
- `edit_level`
- 可选：`views`、`clicks`、`engagement`

### 15.3 投研最小回流字段
- `selected`
- `watch_status`
- `manual_rating`
- `actual_return_5d`
- `actual_return_20d`
- `max_drawdown_after_signal`
- `decision_note`

### 15.4 周度回流任务
每周运行一次统计脚本，输出：
- 选题采纳率 TOP 规则
- 被人工驳回最多的选题模式
- 筛股后 5 日表现统计
- 命中率较差的规则阈值
- 人工高频修订的 AI 输出模式

### 15.5 Act 动作落地
周度统计脚本的输出将直接触发 Act 动作，例如：
- 调整热点聚类的相似度阈值
- 更新选题评分权重
- 修改筛选规则的 PE / ROE 边界
- 更新交易规则引擎的风险阈值
- 对“高成本 + 低价值”的 AI 调用环节启动代码化 / 模板化重构评估

所有调整记录写入 `rule_versions` 表，并保留回滚能力。

### 15.6 原则
- 先做简单统计，不引入复杂机器学习
- 先让系统“能学到什么有效”，再讨论自动学习

## 18. 最终推荐方案

### 推荐级别最高：方案 A（混合架构）

#### 组成
- Python / FastAPI 负责业务逻辑与 HTTP API
- n8n 负责调度
- YMOS 负责投研闭环
- 云端模型负责深度分析与终稿
- 本地模型负责批处理与降本

#### 优点
- 成本与质量平衡最好
- 易于逐步落地
- 可维护性高
- 后续扩展空间大

#### 适合你当前阶段的原因
- 你已有 YMOS 基础
- 你有可用的本地 GPU，但不适合让本地模型扛全部核心任务
- 你同时要做内容生产与投研，必须强调架构稳定性和模块职责划分

## 20. 关键设计结论（供快速决策）

1. **总架构选混合，不选纯云，也不选纯本地。**
2. **投研主脑仍以 YMOS + 云端深度模型为核心。**
3. **本地模型是过滤器、草稿机、RAG 初稿器，不是终极裁判。**
4. **财经自媒体默认半自动，外发前保留人工审校。**
5. **n8n 负责编排，不负责复杂业务逻辑。**
6. **Python 服务必须接口化，避免脚本散落执行。**
7. **MVP 只先闭环一条主线。**
8. **预算、熔断、DLQ、可观测性是架构内建能力，不是后补功能。**
9. **必须有极简审校后台和反馈回流字段。**
10. **先做 MVP 跑通，再做本地模型降本与高级自动化。**
